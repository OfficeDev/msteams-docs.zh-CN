# <a name="build-a-great-app-details-page"></a>生成出色的应用详细信息页面

详细信息页面向用户呈现应用的第一印象。 详细信息页面的每个元素都可用于传达你的愿景和驱动器下载，考虑你想要如何在有限的空间中展示你的应用。 下面是一些提示和技巧，可帮助你在用户安装应用之前与用户互动。

> [!NOTE]
> 确保你的应用信息遵循 [我们的 AppSource 指南来创建有效的应用商店一览](/office/dev/store/create-effective-office-store-listings)。

## <a name="app-name"></a>应用名称

> [!div class="checklist"]
>
> * 应用名称在用户如何在 AppSource 应用商店中发现它中扮演关键角色。 你的应用的短名称将显示在详细信息页面上。
>* 应用名称应反映你的应用，而不引用 Microsoft 或 Microsoft 产品。
>

> **注意**：如果你的应用是 Microsoft 的官方合作关系，则第三方应用的名称需要首先，例如，Microsoft Teams 的 *Salesforce 连接器*。

> [!div class="checklist"]
>
>* 使用这些资源提供指导：

* [应用名称指南](#app-name)
* [Microsoft 商标和品牌准则](https://www.microsoft.com/en-us/legal/intellectualproperty/trademarks/usage/general)

**需要：**

* 选择一个简单易记的名称，该名称将提示你的应用所应做操作。
* 独特。
* 如果需要，请使用 Microsoft 365 引用，而不是 Office 365。

**请勿：**

* 不要忽略空格、大小写不正确或应用名称中包含语言错误。
* 请勿使用类似于现有应用的一般术语或名称。
* 不要在应用名称中使用"Teams"、"Microsoft"、"现有/即将推出的 Microsoft 产品名称"或"应用"。
* 请勿使用括号包含 Microsoft 产品，例如，适用于 Microsoft Teams (*Your-App-Name) 。*

## <a name="short-description"></a>简短说明

这是应用的简要摘要。 不妨原创有吸引力且面向目标受众的说明。 理想情况下，请尝试用一个句子向用户描述你的解决方案及其价值。

**需要：**

* 首先放置最重要的信息。
* 包括客户可能搜索的关键字。
* 如果你需要提及 Microsoft Teams，Microsoft Teams 的第一个提及应完整地写成 *Microsoft Teams*。 如果在同一描述中再次提到 Teams，可以将名称缩短为 *Teams*。
* 对 Microsoft 或 Microsoft Teams 的任何引用都可以作为说明的一部分，并且应遵循 Microsoft 的品牌标准和指南。
* 所有说明的语法必须正确，且没有语言错误。
* 避免使用不必要的大写，例如，说明"用户"而不是"用户"。

**请勿：**

* 不要重复标题。
* 不要将 Microsoft 缩写为"MS"或"MSFT"。
* 请勿使用行话或专用术语-不能假定用户知道要查找什么。
* 除非绝对必要，否则避免不必要的 Microsoft 产品名称引用。
* 请勿指示或暗示应用是 Microsoft 提供的产品/服务。
* 请勿使用自己没有的受版权保护的品牌名称。
* 请勿以短名称使用"for Teams"。

## <a name="long-description"></a>较长说明

> [!div class="checklist"]
>
>* 这提供了一个极具吸引力的叙述性内容，重点介绍解决方案的主要功能、解决方案所解决的问题和目标受众。 通过传达应用的独特功能，用第一句吸引受众。 说明必须不足 4000 个字符;大多数用户只能阅读 300 到 500 个单词。
>* 允许使用什么？

* `<your_app>`  "使用 Microsoft Teams"
* `<for users>`  "使用 Microsoft Teams"
* `<for tasks>`  "在 Microsoft Teams 中"
* `<an app>`  "for Microsoft Teams"
* `<your_app>`  "与 Microsoft Teams 集成"
* "...与 Microsoft Teams 集成"
* "...构建于..."
* "...运行..."
* "...enabled by..."
* "...开发用于..."
* "...专为..."

> **注意**：上述条款也适用于 Microsoft 365 的使用。 Office 365 现在称为 Microsoft 365。 请更新应用说明以反映这一点。

>[!IMPORTANT]
> 请务必将你在 AppSource 条目中撰写的说明精确复制到应用清单中，这些值必须匹配。 Microsoft Teams 将仅使用你在应用清单中提供的说明。

**需要：**

* 使用 [Markdown 格式](https://support.office.com/article/use-markdown-formatting-in-teams-4d10bd65-55e2-4b2d-a1f3-2bebdcd2c772) 来阐明你的说明。  
* 列出有助于读者扫描说明的功能。
* 使用活动语音并直接与用户通话。
* 使用项目符号点列出您的功能。
* 包括帮助或支持链接，以便你的用户知道在有疑问时如何联系你。
* 确保 Microsoft Teams 的第一个提及完整地写成 *"Microsoft Teams"。* 如果稍后在同一描述中再次提到 Teams，可以将名称缩短为 *"Teams"。*
* 仅在必要时才 (Microsoft 或 Microsoft Teams) 可能是详细说明的一部分，并且应遵循 Microsoft 的品牌标准和指南。
* 所有说明的语法必须正确，且没有语言错误。
* 避免对描述中的术语使用不必要的大写 (例如：说明"用户"而不是"用户"。
* 避免使用首字母缩略词。
* 请确保调用限制、帐户依赖关系、配置设置、版本中的未来更新或任何使用限制

>[!NOTE]
> Microsoft Teams 支持以下 Markdown 语法：  
> **链接**。 `[title](url/address/here)`.  
>**图像** `![alt text](url/address/here)` . .  
> **加粗**。 `**bold text**`   `__bold text__`.  
> **斜体**。 `*italicized text*`  `_italicized text`.  
>**[已排序列表](https://www.markdownguide.org/basic-syntax/#ordered-lists)**<br>
>`1. first` 
<br>` 1. second ` 
<br>`1.third`<br>
>**[无序列表](https://www.markdownguide.org/basic-syntax/#unordered-lists)**<br>
` - short` <br>`- bulleted` <br>`- list`<br>
>**Newline**。 使用 `\n` 字符指定新行。
 >**转义。** 使用内联反线转义特殊字符。 `\*asterisk`.

**请勿：**

* 请勿在描述中放入太多关键字，这令人分心，并且对应用的可发现性没有帮助。
* 请勿短名称使用 *"Teams"* 或 *"Microsoft Teams"。*
* 除非绝对必要，否则避免不必要的 Microsoft 产品名称引用。
* 不要指示该应用是 Microsoft 提供的产品/服务。
* 请勿使用自己没有的受版权保护的品牌名称。

* 合作伙伴不得使用或模仿任何 Microsoft 标记，或在标记内使用任何 Microsoft 产品或服务的名称。

## <a name="screenshots"></a>屏幕截图

在合作伙伴中心上载的屏幕截图显示在[](https://partner.microsoft.com) [AppSource](https://appsource.microsoft.com/marketplace/apps?product=office%3Bteams&page=1)和 Teams 客户端的应用一览中。 它们提供应用的可视预览以及应用说明。
你可以提供一到五张格式设置为 .png、.jpg 或 .gif 文件的屏幕截图。 屏幕截图应为 1366 x 768 像素，最大大小为 1024 KB。

**需要：**

* 专注于突出显示应用的所有功能。
* 内容应准确地表示你的应用。
* 文本应填充良好且不会过多。
* 你可以将屏幕截图用背景色环绕，并添加类似于 [Freshdesk](https://appsource.microsoft.com/product/office/WA104381505?src=office&tab=Overview) 示例的营销内容;但是，尺寸不会仅包含屏幕截图，而是包括整体图像。

<img width="800px" alt="Freshdesk screenshot" src="../../../../assets/images/freshdesk.png" />

**请勿：**

* 请勿显示特定设备，如手机或笔记本电脑。
* 不要在应用外部显示任何部件版式/UI。
* 请勿在屏幕截图中捕获任何 Teams 或浏览器 UI。
* 不要包含不精确反映应用实际 UI（如显示网站而不是 Teams 选项卡）的模拟。

有关更多最佳做法， *请参阅：* 精心制作 [有效的 AppSource 应用商店图像](/office/dev/store/craft-effective-appsource-store-images)。

## <a name="videos"></a>视频

如果图像值得一千字，则视频值得一千个图像。 视频是传达使用你的应用的好处最有效的方法。 它将放置在应用详细信息页面上所有屏幕截图的前面。 请务必提到以下内容：

* 应用的工作原理。
* 使用你的应用可实现的目标。
* 使用应用的好处。
* 你的目标用户。

请记住，保持演示文稿的简短和友好 - 介于 30 到 90 秒之间。
